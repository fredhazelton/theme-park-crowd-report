## Summary

Add a daily auto-generated blog post to hazeydata.ai that compares the previous day's predicted WTI against observed WTI for Walt Disney World's four parks. The post surfaces the biggest prediction miss, breaks it down to entity-level (which attractions drove the gap), and detects potential unannounced closures from wait time data gaps.

This is a **content product** built on top of the WTI pipeline — it turns our daily prediction accuracy into public-facing analysis that theme park fans want to read.

---

## The Product

**Name:** WDW Daily Recap
**URL pattern:** `hazeydata.ai/theme-park-crowd-report/blog/wdw-daily-recap-YYYY-MM-DD.html`
**Frequency:** Daily, published after the observed WTI tweet (~8:30 AM ET)
**Scope:** Walt Disney World only (MK, EP, HS, AK) — Phase 1
**Format:** Pure data/template (no LLM generation) — Phase 1

### What It Answers

For theme park fans following @DisneyStatsWhiz who saw yesterday's predicted WTI tweet and this morning's observed WTI reply:

1. **How did our predictions hold up?** — Park-level predicted vs observed, with color-coded deltas
2. **Where did the miss come from?** — Entity-level breakdown for the park with the biggest miss. Was it one headliner or across the board?
3. **Were there closures?** — Detect unannounced ride closures from gaps in wait time data. When a headliner closes, neighboring queues spike — this is often the explanation for big misses.

### What It Does NOT Do (Phase 1)

- Weather analysis (Phase 3 — needs new data source)
- LLM-written narrative (Phase 2 — add after template is proven)
- Multi-property coverage (future — after WDW template is stable)
- Trend analysis across multiple days (future — needs accumulation)

## Data Sources

All data required for Phase 1 already exists in the pipeline. No new data collection needed.

### 1. Park-Level WTI (Predicted vs Observed)

**Source:** Content JSONs generated by `s14_content.py`
**Location:** `/home/wilma/hazeydata/pipeline/content/`
**Files:**
- `wti_predicted_{date}.json` — predicted WTI per park, generated at pipeline time
- `wti_observed_{date}.json` — observed WTI per park, generated next morning

These are the exact same data that power the tweets. The recap just displays them in more detail.

### 2. Entity-Level Accuracy

**Source:** Accuracy archive + fact tables
**Location:** `/home/wilma/hazeydata/pipeline/accuracy/` and `/home/wilma/hazeydata/pipeline/fact_tables/parquet/`

For the park with the biggest WTI miss, query:
- Predicted wait per entity (from archived forecast for the reference date)
- Observed actual wait per entity (from fact tables, `wait_time_type = 'ACTUAL'`)
- Delta per entity, sorted by absolute error descending

This uses the same methodology as `s10_accuracy.py` — ACTUAL wait_time_type, TIME_BUCKET with midpoint rounding.

### 3. Closure Detection

**Source:** Fact tables (POSTED wait times)
**Location:** `/home/wilma/hazeydata/pipeline/fact_tables/parquet/`

An unannounced closure appears as a gap in POSTED wait time observations for an entity. The detection algorithm:

```
For each entity at the "biggest miss" park:
  1. Get all POSTED observations for the reference date, sorted by time
  2. Find gaps > 30 minutes between consecutive observations
  3. If gap > 60 minutes AND entity was posting before AND resumed after:
     → Flag as "likely unannounced closure"
  4. Report: entity name, gap start time, gap end time, gap duration
```

A 30-minute gap is normal (observation frequency varies). A 60+ minute gap during operating hours is almost certainly a closure. When a headliner like Flight of Passage closes for 2 hours, guests redistribute to other AK rides — that's exactly the kind of event that creates a WTI miss.

### 4. Entity Names

**Source:** `dimension_tables/dimentity.csv`
**Rule (from V4 Design):** Every report that shows entity codes MUST also show entity names. "AK01 — Expedition Everest: +12.3 min" not just "AK01: +12.3 min".

### Evaluation Methodology

Entity-level accuracy uses **identical methodology to `s10_accuracy.py`**:
- `wait_time_type = 'ACTUAL'` for observed waits
- `TIME_BUCKET(INTERVAL '5 minutes', ts + INTERVAL '2 minutes 30 seconds')` for time slot alignment
- Synthetic actuals as fallback
- Entity names from `dimentity.csv`

This is non-negotiable. The recap must use the same evaluation as the pipeline. If `shadow_evaluate.py` taught us anything, it's that divergent evaluation methodology creates confusion.

### Pattern Detection

The `pattern` field categorizes how the miss is distributed:

- `"concentrated"` — Top entity accounts for >40% of total park error
- `"top_heavy"` — Top 3 entities account for >60% of total park error
- `"distributed"` — Error spread across many entities, no single dominant source

The `pattern_detail` is a template-filled sentence, not LLM-generated.

### Closure Detection

For each entity at the spotlight park, scan POSTED observations:

```sql
WITH obs AS (
    SELECT entity_code, observed_at_ts,
           LAG(observed_at_ts) OVER (PARTITION BY entity_code ORDER BY observed_at_ts) as prev_ts
    FROM fact_tables
    WHERE park_date = '{date}' AND wait_time_type = 'POSTED'
      AND entity_code LIKE '{park_code}%'
)
SELECT entity_code,
       prev_ts as gap_start,
       observed_at_ts as gap_end,
       DATEDIFF('minute', prev_ts, observed_at_ts) as gap_minutes
FROM obs
WHERE DATEDIFF('minute', prev_ts, observed_at_ts) > 60
ORDER BY gap_minutes DESC
```

## Publish Script Specification

`scripts/daily_recap_publish.py` runs on Dino's Mac Mini.

### Steps

1. **SSH to wilma-server:** Run `extract_daily_recap.py` to get recap JSON
2. **SCP the JSON** to Mac Mini (or read via SSH stdout)
3. **Render HTML** from recap JSON using the blog template
4. **Clone/pull `hazeydata.ai` repo** on Mac Mini
5. **Write HTML file** to `theme-park-crowd-report/blog/wdw-daily-recap-YYYY-MM-DD.html`
6. **Update `index.html`** — prepend the new post to the blog listing
7. **Git commit + push** to `hazeydata.ai` repo (master branch)
8. **Post to Discord #wti-pipeline:**
   ```
   📊 WDW Daily Recap — March 30, 2026
   MK 15.6→18.5 (+2.9) 🟢 | EP 21.8→26.2 (+4.4) 🟡 | HS 26.0→31.7 (+5.7) 🟡 | AK 24.1→39.2 (+15.1) 🔴
   Spotlight: Animal Kingdom — Flight of Passage drove the miss (+37 min)
   https://hazeydata.ai/theme-park-crowd-report/blog/wdw-daily-recap-2026-03-30.html
   ```

### Blog Index Update

The publish script must update `theme-park-crowd-report/blog/index.html` to include the new post. The existing index uses a static HTML list of posts. The script should:

1. Read current `index.html`
2. Find the insertion point (after the blog listing header, before the first post card)
3. Insert a new post card HTML block
4. Write the updated file

To avoid index.html growing unbounded, only keep the 30 most recent posts visible. Older posts remain accessible via direct URL but drop off the index page.

## Key Rules

1. **Evaluation methodology must match `s10_accuracy.py` exactly.** No divergent SQL. The extract script uses ACTUAL wait_time_type, TIME_BUCKET with midpoint rounding, and synthetic actuals fallback. This was established as a hard rule in Session 25.

2. **Entity names are mandatory.** Every entity code must display alongside its human-readable name. "AK10 — Avatar Flight of Passage" not "AK10". (V4 Design display rule.)

3. **Proof batch first.** Before the cron goes live, generate the recap for 3-5 historical dates and review the HTML output. (Rule 17.)

4. **Blog index stays manageable.** Cap visible posts at 30 on the index page. Direct URLs work forever.

5. **No LLM in Phase 1.** Template-driven sentences only. The pattern detection (`concentrated` / `top_heavy` / `distributed`) drives the summary line. LLM narrative is Phase 2.

6. **Publish timing: 9:00 AM ET.** After the observed tweet (8:30 AM) so the recap links to the same data the tweet reported.
